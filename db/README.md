# Database Strategy (P4.1)

ORM choice: Prisma (Node 18 compatible release line)

Why Prisma:

- Stable PostgreSQL support and migration workflow
- Clear schema for iterative domain modeling
- Easy onboarding for future repository-layer transition

Structure:

- `db/prisma/schema.prisma`: source of truth schema
- `db/prisma/migrations/0001_init/migration.sql`: initial SQL migration baseline
- `docker-compose.yml`: local PostgreSQL runtime
- `scripts/check-db-connection.mjs`: connection verification for local/staging

Phase intent:

- Phase 4.1 only defines schema/migration project scaffolding.
- Runtime API still uses JSON files until Phase 4.5/P4.6 cutovers.

Environment setup commands:

- `npm run db:up`
- `npm run db:check:local`
- `npm run db:check:tunnel`
- `npm run db:generate`
- `npm run db:migrate:dev`
- `npm run db:import:dry`
- `npm run db:import`
- `npm run db:verify:json`
- `npm run db:verify`
- `npm run db:verify:tunnel`
- `npm run db:verify:staging`

Verification report output:

- `docs/DATA_VERIFICATION_REPORT.md` is generated by verification scripts.
- Use `db:verify:json` when DB is not available to snapshot expected JSON counts.
- Use `db:verify` after import to compare JSON vs PostgreSQL counts and sampled records.

Staging credential contract:

- `STAGING_DATABASE_URL` is required for `npm run db:check:staging`
- Store real staging values in secret manager, not git

Tunnel contract:

- `TUNNEL_DATABASE_URL` can target an SSH-forwarded PostgreSQL endpoint (for example `127.0.0.1:18789`).
- Use this when Docker is unavailable but a remote DB is reachable through SSH local forwarding.
